# 冻结策略与验收标准

## 冻结模块

以下区域冻结，除非五条选中链路无法编译或无法跳转，需要极小适配：

- 登录、注册、密码、验证码、输入页
- 个人资料、用户详情，但聊天入口按钮路由除外
- 支付、订单、购物车、商品、地址
- 地图、定位
- 视频、广告、直播、腾讯播放器
- 分享、MobSDK
- 推送，但聊天通知路由除外
- 引导、启动、扫码、Web、设置、关于
- 房间、语音、其他演示模块
- 老通用 UI 工具，除非选中链路需要窄桥接

## 冻结区允许改什么

允许：

- 把点击入口路由到新的选中链路 Activity/Compose host。
- 添加很薄的 adapter/facade 调用新 Kotlin 代码。
- 修复依赖或 SDK 升级导致的编译错误。
- 保持旧功能能启动、能跳转、能基本运行。

不允许：

- 把冻结页面改成 Compose。
- 移动冻结模块包结构。
- 替换冻结模块 SDK。
- 做纯样式、纯命名、纯整理型重构。
- 删除冻结功能入口，除非它阻塞 App 启动且没有低成本修复。

## 兼容策略

选中链路可以引入新的 Kotlin Service/Repository，同时保留旧 Java 调用方。边界处使用桥接类：

- Java Activity/Fragment -> 新 Compose host 或兼容门面。
- 旧 Manager interface -> 新 Repository/Controller。
- 第三方 callback SDK -> Kotlin Flow/suspend 包装。
- 旧事件通知 -> 新 SharedFlow/StateFlow；只有遗留屏幕还在监听时，才临时保留 EventBus 桥。

## 验收级别

用户明确要求“能跑就行，不做测试”。因此本轮验收是：

- 构建成功。
- App 能安装或从 Android Studio/Gradle 启动。
- 五条选中链路通过人工冒烟。
- 冻结模块只做启动/跳转级别的基本确认。

## 非目标

- 不要求自动化测试覆盖。
- 不要求像素级 UI 一致。
- 不对冻结模块做完整功能回归。
- 不改后端/API。
- 不做产品行为重设计。

## 风险登记

| 风险 | 区域 | 缓解 |
|---|---|---|
| 第三方 SDK callback 难以干净包装 | 聊天 | 保留 Java 兼容门面，逐步把状态移动到 Kotlin Flow |
| 媒体通知/系统控制行为回归 | 音乐 | 先在 facade 后接 Media3，冒烟确认控制链路再替换旧 Manager |
| Gradle 升级导致第三方 SDK 不兼容 | 构建 | 小步升级，冻结无关 SDK 替换 |
| Compose/XML 共存混乱 | UI | 只做选中链路 host，不做全局导航重写 |
| 上传/下载进度过度发射 | Feed/Download | UI 状态节流，列表使用稳定 key |

## 暂停条件

出现以下情况要暂停并重新评估：

- 必需第三方 SDK 不兼容现代 Gradle/Kotlin 基线。
- 选中链路需要后端配合改接口。
- 依赖升级后 App 无法启动，失败点在冻结 SDK。
- 替换一条选中链路会连带重写多个冻结领域。
